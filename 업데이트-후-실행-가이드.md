# OMC 업데이트 후 패치 다시 적용하기

OMC를 새 버전으로 올리면 `git-master` 같은 플러그인 파일이 새 캐시 폴더에
새로 깔린다. 그때 우리가 넣어둔 라우팅 패치는 함께 사라진다. 이 문서는
업데이트한 뒤 패치를 다시 얹는 방법을 정리한 것이다.

세션이 켜질 때마다 검사하면 시작이 느려진다. 그래서 자동화 대신, 업데이트한
뒤 직접 한 번 실행하는 방식으로 둔다.

## 언제 실행하나

- OMC를 업데이트한 직후. `/oh-my-claudecode:omc-setup`을 돌렸거나 플러그인
  버전이 올라갔을 때.
- `git-master`가 프로젝트의 git 스킬을 무시하고 평소와 다르게 굴 때. 패치가
  날아갔을 가능성이 있다.

## 실행 방법

먼저 무엇이 바뀌었는지 보고, 상태를 확인하고, 적용하고, 충돌이 났을 때만 손을 댄다.

### 0. 업스트림이 뭘 바꿨는지 본다 — 파일을 바꾸지 않는다

```bash
~/Documents/LFin/my-project/claude-skills/claude-overlay/upstream-changes.sh
```

우리가 패치한 파일들(git-master, planner 등)이 업데이트 사이에 업스트림에서
어떻게 바뀌었는지, marketplace의 git 이력으로 커밋 메시지와 변경 통계를 보여준다.
어떤 패치가 깨끗이 붙고 어떤 게 충돌날지 미리 가늠하는 단계다. 변경이 없으면
"no upstream changes"가 뜨고, 있으면 그 커밋과 diff를 근거로 뒤의 충돌을 푼다.
캐시에서 생성되는 파일(예: `skill-bodies/`)은 git 이력이 없어 그 자리에서
"not tracked"로 표시되고, 변경은 머지 단계에서 드러난다.

### 1. 상태 확인 — 파일을 바꾸지 않는다

```bash
~/Documents/LFin/my-project/claude-skills/claude-overlay/apply.sh
```

기본은 dry-run이라 아무것도 건드리지 않고 현재 상태만 보여준다. 줄 앞의
라벨로 읽으면 된다.

- `OK` — 이미 패치돼 있음. 할 일 없음.
- `CLEAN` — 패치가 빠져 있고, 깨끗하게 적용할 수 있음.
- `CONFLICT` — 업스트림이 우리가 고친 부분을 같이 건드림. 사람 판단이 필요함.
- `MISSING` — 패치 대상 파일이 어느 루트에도 없음. 업스트림이 그 파일을 지웠거나
  이름을 바꿨다는 뜻이고, 우리 커스터마이즈가 더 이상 적용되지 않는다. 새 버전에서
  어디로 갔는지 찾아 타깃을 바꾸거나, 폐기된 거면 그 패치를 정리한다.

`OK`만 떴다면 여기서 끝이다.

### 2. 적용

`CLEAN`이 보이면 `--write`로 실제 적용한다.

```bash
~/Documents/LFin/my-project/claude-skills/claude-overlay/apply.sh --write
```

이때 합쳐진 결과는 **라이브 플러그인 파일에만** 쓰인다. `baseline.md`와
`patched.md`는 건드리지 않는다. 즉 `--write`만으로는 baseline이 갱신되지 않고,
처음 패치했던 버전에 계속 고정돼 있다.

업스트림이 그 파일을 손봤지만 우리 패치와 겹치지 않은 경우, 출력에
`upstream drifted, merged cleanly`가 뜨고 끝에 이런 리마인더가 붙는다.

```
REMINDER: upstream drifted on: git-master
          baseline is still pinned to the old version. Re-run with --update-baseline ...
```

이건 "라이브 파일은 잘 적용됐지만 baseline이 옛 버전에 머물러 있다"는 신호다.
이 상태로 두면 당장은 멀쩡해도, 업스트림이 그 주변을 계속 고치면서 낡은
baseline의 맥락이 어긋나 나중에 헛충돌이 날 수 있다. 그래서 드리프트가 났을
때는 baseline을 그 버전으로 당겨두는 편이 낫다.

```bash
~/Documents/LFin/my-project/claude-skills/claude-overlay/apply.sh --write --update-baseline
```

이러면 `baseline.md`가 새 원본으로, `patched.md`가 합친 결과로 갱신돼 다음
업데이트 때 짧은 차이만 다시 머지하면 된다. 드리프트가 없으면(`upstream
unchanged`) baseline은 이미 같으니 `--update-baseline`을 붙일 필요가 없다.

> 정리하면 baseline이 갱신되는 시점은 단 하나다 — **드리프트가 났고
> `--update-baseline`을 줬을 때.** 그 외에는 그대로 둔다.

### 3. 충돌이 났을 때

`CONFLICT`는 업스트림이 우리 패치와 같은 줄을 고쳤다는 뜻이다. 스크립트는
자동으로 덮어쓰지 않고 멈춘다. 대신 target 파일 옆에
`git-master.md.merge-conflict`를 남기는데, 그 안에 표준 충돌 표시가 들어 있다.

```
<<<<<<< upstream-new      새 업스트림이 바꾼 내용
||||||| baseline          우리가 패치하기 전 원본
=======
>>>>>>> my-patch          우리 패치
```

업스트림 변경과 우리 라우팅을 어떻게 합칠지 정한 뒤, 충돌 표시를 지우고 그
결과를 원래 target 파일에 덮어쓴다. 그다음 재적용 때 깨끗하게 맞물리도록
`patches/git-master/baseline.md`를 새 원본으로, `patched.md`를 합친 결과로
바꿔 둔다. baseline을 다시 만드는 방법은 README에 적어 두었다.

## 편하게 부르기 (선택)

매번 긴 경로를 치기 번거로우면 `~/.zshrc`에 짧은 별칭을 둔다.

```bash
alias claude-overlay='~/Documents/LFin/my-project/claude-skills/claude-overlay/apply.sh'
```

새 셸을 열면 `claude-overlay`로 확인, `claude-overlay --write`로 적용이다.

## 한눈에

| 상황 | 명령 |
|------|------|
| 업데이트 후 상태 확인 | `apply.sh` |
| 패치 재적용 (드리프트 없음) | `apply.sh --write` |
| 패치 재적용 + baseline 갱신 (드리프트 있음, 권장) | `apply.sh --write --update-baseline` |
| 충돌 | `.merge-conflict`를 직접 해결한 뒤 baseline 다시 만들기 |

권장 흐름은 단순하다. 먼저 `apply.sh`로 보고, `OK`만이면 끝. `CLEAN`이나
`REMINDER`에 드리프트가 보이면 `--write --update-baseline`. `CONFLICT`면 손으로
해결. 헷갈리면 드리프트가 보일 때 그냥 `--update-baseline`을 붙이면 된다 —
드리프트가 없으면 그 옵션은 아무 일도 하지 않으니 손해가 없다.

종료 코드는 스크립트로 엮을 때 쓴다. `0`이면 정상, `2`면 충돌 하나 이상,
`3`이면 머지 오류, `4`면 어떤 패치의 대상 파일이 모든 루트에서 사라진 경우다.
